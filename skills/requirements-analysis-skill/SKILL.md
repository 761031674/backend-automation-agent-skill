# 需求分析技能

本 Skill 提供一套标准化的需求分析工具集，帮助 Agent 系统化地处理业务需求，将模糊描述转化为结构化、可执行的需求文档。

## 🛠️ 核心工具

### 1. 逻辑冲突检测矩阵
在分析需求时，对照以下维度逐项检查：

| 检测维度 | 检查项 | 示例 |
|---------|--------|------|
| **状态流转** | 是否存在未定义的中间状态？ | 营销员从"在职"到"离职"是否有中间态？ |
| **数据权限** | 是否考虑了跨组织/跨角色的数据隔离？ | ME Head 是否能看到下属 ME 的数据？ |
| **时序依赖** | 是否依赖尚未发生的外部事件？ | 推送时间是否必须在数据同步完成后？ |
| **数据一致性** | 跨系统数据是否有一致性保证？ | 核心系统离职状态与 CLMS 是否同步？ |
| **边界条件** | 空值/极端值是否有处理？ | 分页查询结果为空时展示什么？ |

### 2. AC 编写模板 (Given-When-Then)
针对每个功能点编写验收标准：

- **Given**: 初始上下文（如：营销员处于离职状态）
- **When**: 触发动作（如：ME 点击邀约面谈按钮）
- **Then**: 预期结果（如：弹出提示框告知该人员已离职，且禁止创建面谈活动）

**规则**：
- 每个功能点至少包含 1 个正向用例 + 1 个反向用例
- 边界条件必须单独成条（如：日期起止相等、搜索结果为空）
- 异常场景必须定义明确的错误提示信息

### 3. 非功能性需求检查表
- [ ] **性能**：核心接口响应时间是否在 500ms 以内？大数据量场景是否支持分页/游标？
- [ ] **安全**：是否涉及 PII（如手机号、身份证号）泄露风险？是否需要脱敏展示？
- [ ] **兼容性**：是否与旧版客户端/API 版本兼容？
- [ ] **可用性**：是否考虑了降级方案？关键链路是否有熔断/限流？
- [ ] **数据合规**：是否符合数据保留和删除策略？

### 4. 需求完整性审查清单
- [ ] 已明确功能范围（做什么 + 不做什么）
- [ ] 已识别所有用户角色和权限边界
- [ ] 已定义输入/输出数据结构及字段规则
- [ ] 已明确业务规则（枚举值、计算公式、过滤条件）
- [ ] 已定义异常场景及处理方式
- [ ] 已明确数据来源和同步机制
- [ ] 已确认与现有功能的交互关系（联动/冲突）

## 📋 工作流程

### 第一步：需求拆解
将大需求拆解为原子化的功能点，每个功能点应满足：
- 可独立测试
- 有明确的输入和输出
- 有清晰的业务价值

### 第二步：逻辑质疑
对每个功能点提出"如果...会怎样"的假设，至少覆盖：
1. 数据为空/不存在
2. 边界值（最大值、最小值、临界点）
3. 并发/时序冲突
4. 权限不足
5. 外部依赖失败

### 第三步：AC 定义
使用 AC 模板为每个功能点定义成功与失败的判定标准：
- **正向用例**：Given [初始状态], When [操作], Then [预期结果]。
- **反向用例**：覆盖权限不足、数据为空、非法输入等场景。
- **边界条件**：单独列出日期临界点、数值极限、分页空值等。

### 第四步：风险标记
对以下类型的问题标记为 🔴（阻塞）或 🟡（警告）：
- 🔴 **阻塞项**：需求缺失导致无法实现、逻辑死锁、严重安全漏洞。
- 🟡 **警告项**：需求模糊但可基于假设推进、非功能需求待确认。

## ⚠️ 异常场景处理

### 场景1：需求描述过于简略
**处理**：
1. 基于 5W1H 框架列出澄清问题
2. 提供假设性方案供用户选择
3. 明确标注"待确认"部分

### 场景2：发现与现有架构/规范冲突
**处理**：
1. 说明冲突的具体表现和影响范围
2. 提供至少 2 个备选方案（A. 调整需求 B. 调整架构 C. 引入适配层）
3. 推荐方案并说明理由

### 场景3：非功能性需求缺失
**处理**：
1. 基于业务场景推断合理的非功能要求
2. 提供行业参考值（如金融系统接口响应时间 < 500ms）
3. 标记为"建议补充"而非强制
