# 并行分派规范

## 何时使用

**适用场景：**
- 3个以上测试文件因不同根因失败
- 多个子系统独立出现故障
- 每个问题无需其他问题的上下文即可理解
- 排查之间无共享状态

**不适用场景：**
- 失败是相关的（修复一个可能修复其他的）
- 需要理解完整的系统状态
- 智能体之间会互相干扰

## 决策流程

```
存在多个失败? → 是 → 它们是否独立? → 是 → 能否并行工作? → 是 → 并行分派
                          ↓否                ↓否
                   单个智能体排查      顺序执行智能体
```

## 分派前自检清单

开始并行分派前，先完成以下确认：

- [ ] **失败已量化**：已统计失败数量和类型，确认 ≥3 个独立失败（不足 3 个建议顺序处理）
- [ ] **关联性已排除**：快速扫描错误信息，未发现共同根因（如统一配置错误、同一上游服务故障）
- [ ] **资源已评估**：当前可用并行 slot ≥ 需要的分派数量（最大并行 ≤5，避免资源耗尽）
- [ ] **上下文已收集**：每个失败的关键日志、涉及文件、预期行为已提取，能独立描述每个问题

> **自检结果**：全部勾选 → 进入识别问题域；任一未勾选 → 先补充信息再分派

## 工作流程

### 前置检查

**输入**：失败列表（包含错误日志、测试报告、涉及文件路径）
**动作**：
1. 统计失败数量和类型（<3个时建议顺序处理而非并行）
2. 快速扫描错误信息，判断是否有相同根因（如都是同一配置问题导致）
3. 如果失败高度相关 → 直接进入单一智能体排查流程

**输出**：
- 失败数量统计（N个）
- 关联性评估结果（独立/相关）
- 建议的分派方案（X个并行任务）

> **检查点**：向用户确认"失败数量共 N 个，建议拆分为 X 个并行任务，是否继续？"

### 1. 识别独立的问题域

**输入**：失败列表、错误日志、测试报告
**输出**：独立问题域分组方案（JSON格式或表格）

按故障分组示例：
```
组1: agent-tool-abort.test.ts → 工具中止逻辑
组2: batch-completion-behavior.test.ts → 批量完成行为
组3: tool-approval-race-conditions.test.ts → 竞态条件处理
```

分组原则：
- 同一子系统/模块的失败归为一组
- 涉及相同代码文件的失败归为一组
- 错误信息中包含相同关键字的失败可能相关

> **检查点**：向用户展示分组方案并确认"这些失败确实是独立的吗？如有相关请指出"

### 2. 创建聚焦的智能体任务

**输入**：问题域分组方案
**输出**：智能体任务列表（每个任务包含以下必填字段）

每个智能体获得的结构化任务描述：
```markdown
### 任务：{具体目标}

**问题域**：{测试文件/子系统名}
**上下文**：
- 失败信息：{关键错误日志，至少包含错误类型和行号}
- 涉及文件：{文件列表，区分生产代码和测试代码}
- 预期行为：{正确行为描述，使用 Given-When-Then 格式}

**约束条件**：
- 只修改 {指定范围} 内的代码（明确列出允许修改的文件路径）
- 不要修改其他智能体负责的文件（列出禁止修改的文件清单）
- 如果需要修改共享代码，先报告再执行（定义共享代码的判断标准）

**预期输出**：
1. 根因分析（2-3句话，说明失败原因）
2. 修改内容摘要（列出修改的文件、行号、修改前后对比）
3. 验证结果（测试是否通过，如未通过说明原因）
```

> **检查点**：向用户确认"这些任务范围和约束是否合适？是否需要调整？"

### 3. 并行分派

```
Task("修复 agent-tool-abort.test.ts 的失败")
Task("修复 batch-completion-behavior.test.ts 的失败")
Task("修复 tool-approval-race-conditions.test.ts 的失败")
// 三个任务并发运行
```

> **检查点**：向用户确认"是否开始并行执行？"

### 4. 审查与集成

**输入**：各智能体返回的修复结果
**输出**：集成后的修复方案

- 阅读每个总结
- 验证修复之间没有冲突：
  1. **文件重叠检查**：列出各智能体修改的文件清单，比对是否有交集 → 有交集则标记冲突文件，进入人工合并流程
  2. **代码冲突检查**：对冲突文件，比对各智能体修改的行号范围 → 同一行被多智能体修改则必须人工合并或重新分配范围
  3. **回归验证**：运行完整测试套件 → 全部通过方可集成；有失败则回滚并重新分派失败任务
- 集成所有更改

> **检查点**：向用户展示修复摘要，确认"这些修改之间没有冲突，可以集成吗？"

## 智能体提示词模板

好的智能体提示词应该包含以下要素：

### 模板结构

```markdown
### 任务：{具体目标}

**问题域**：{测试文件/子系统名}
**上下文**：
- 失败信息：{关键错误日志}
- 涉及文件：{文件列表}
- 预期行为：{正确行为描述}

**约束条件**：
- 只修改 {指定范围} 内的代码
- 不要修改其他智能体负责的文件
- 如果需要修改共享代码，先报告再执行

**预期输出**：
1. 根因分析（2-3句话）
2. 修改内容摘要（列出修改的文件和行号）
3. 验证结果（测试是否通过）
```

### 示例

```markdown
### 任务：修复 agent-tool-abort.test.ts 的失败

**问题域**：src/agents/agent-tool-abort.test.ts
**上下文**：
- 失败信息："Expected 'interrupted at' but got undefined"
- 涉及文件：agent-tool-abort.test.ts, tool-executor.ts
- 预期行为：工具中止时应捕获部分输出

**约束条件**：
- 只修改 tool-executor.ts 中的超时逻辑
- 不要修改测试文件
- 不要修改其他agent的代码

**预期输出**：
1. 根因分析：...
2. 修改内容摘要：...
3. 验证结果：...
```

## 常见错误

| 错误做法 | 正确做法 |
|---------|---------|
| 太宽泛："修复所有测试" | 具体明确："修复 agent-tool-abort.test.ts" |
| 无上下文："修复竞态条件" | 提供上下文：粘贴错误信息和测试名称 |
| 无约束：智能体可能重构所有代码 | 设置约束："不要修改生产代码" |
| 模糊的输出要求："修好它" | 明确要求："返回根因和修改内容的总结" |

## 执行约束

- **最大并行数**：建议不超过 5 个智能体同时运行（避免资源耗尽和上下文混乱）
- **超时设置**：每个智能体任务建议设置 10-15 分钟超时
- **失败回退**：任一智能体失败时，收集已完成的成果，失败任务降级为顺序处理

## 异常处理

| 场景 | 处理策略 |
|------|---------|
| 失败可能相关 | 重新评估关联性；放弃并行改为单一智能体排查 |
| 智能体之间出现冲突 | 1. 检查冲突文件范围和冲突行号<br>2. 优先保留业务逻辑正确的修改<br>3. 手动合并或要求相关智能体重新分配范围 |
| 智能体返回结果质量不足 | 要求补充上下文；转交另一个智能体；单独处理 |
| 资源限制无法并行 | 降级为顺序执行；保持任务聚焦原则 |
| 智能体超时未返回 | 1. 标记该任务为超时<br>2. 收集其他已完成结果<br>3. 对超时任务单独重新分派或人工处理 |
| 智能体修改与现有 PR/未提交更改冲突 | 1. 检查当前工作区是否有未提交更改或进行中的 PR<br>2. 优先保留现有 PR 的修改意图<br>3. 如需覆盖，先提交/暂存现有更改再集成 |
| 所有智能体都修改了同一文件 | 1. 按业务逻辑正确性优先合并<br>2. 如无法自动合并，向用户展示冲突内容请求决策 |

> **项目上下文**：当前项目的微服务列表通过 `project-context` 查阅。

## 实际案例

**案例：多服务独立故障排查**

- 服务A：慢SQL导致接口超时
- 服务B：认证偶发失败
- 服务C：文件上传偶发失败

**分派：**
```
智能体1 → 优化服务A慢SQL（通过 `project-context` 查阅数据库设计规范）
智能体2 → 排查服务B认证配置（通过 `project-context` 查阅架构设计规范）
智能体3 → 调整服务C连接池
```

**结果：** 所有修复互相独立，无冲突，各服务压测通过

## 验证 checklist

- [ ] 已确认失败之间确实独立
- [ ] 每个智能体任务范围清晰、自包含
- [ ] 已设置约束条件（不修改其他智能体负责的文件）
- [ ] 已明确预期输出
- [ ] 集成前检查了修改冲突
- [ ] 运行了完整测试套件验证
