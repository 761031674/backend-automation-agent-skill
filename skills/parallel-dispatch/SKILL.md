# 并行分派规范

## 何时使用

**适用场景：**
- 3个以上测试文件因不同根因失败
- 多个子系统独立出现故障
- 每个问题无需其他问题的上下文即可理解
- 排查之间无共享状态

**不适用场景：**
- 失败是相关的（修复一个可能修复其他的）
- 需要理解完整的系统状态
- 智能体之间会互相干扰

## 决策流程

```
存在多个失败? → 是 → 它们是否独立? → 是 → 能否并行工作? → 是 → 并行分派
                          ↓否                ↓否
                   单个智能体排查      顺序执行智能体
```

## 工作流程

### 1. 识别独立的问题域

**输入**：失败列表、错误日志、测试报告
**输出**：独立问题域分组方案

按故障分组示例：
- 文件A测试：工具审批流程
- 文件B测试：批量完成行为
- 文件C测试：中止功能

> **检查点**：向用户确认"这些失败确实是独立的吗？"

### 2. 创建聚焦的智能体任务

**输入**：问题域分组方案
**输出**：智能体任务列表

每个智能体获得：
- **明确范围**：一个测试文件或子系统
- **清晰目标**：让这些测试通过
- **约束条件**：不修改其他代码
- **预期输出**：发现和修复内容的总结

> **检查点**：向用户确认"这些任务范围和约束是否合适？"

### 3. 并行分派

```
Task("修复 agent-tool-abort.test.ts 的失败")
Task("修复 batch-completion-behavior.test.ts 的失败")
Task("修复 tool-approval-race-conditions.test.ts 的失败")
// 三个任务并发运行
```

> **检查点**：向用户确认"是否开始并行执行？"

### 4. 审查与集成

**输入**：各智能体返回的修复结果
**输出**：集成后的修复方案

- 阅读每个总结
- 验证修复之间没有冲突：
  1. 检查各智能体修改的文件是否有重叠
  2. 检查是否有对同一行代码的冲突修改
  3. 运行完整测试套件验证集成后无回归
- 集成所有更改

> **检查点**：向用户展示修复摘要，确认"这些修改之间没有冲突，可以集成吗？"

## 智能体提示词结构

好的智能体提示词应该是：
1. **聚焦的** - 一个清晰的问题域
2. **自包含的** - 包含理解问题所需的所有上下文
3. **明确输出要求** - 智能体应该返回什么

```markdown
修复 src/agents/agent-tool-abort.test.ts 中 3 个失败的测试：

1. "should abort tool with partial output capture" - 期望消息中包含 'interrupted at'
2. "should handle mixed completed and aborted tools" - 快速工具被中止而非完成
3. "should properly track pendingToolCount" - 期望 3 个结果但得到 0 个

你的任务：
1. 阅读测试文件，理解每个测试验证的内容
2. 找到根因
3. 修复方式：用基于事件的等待替换任意超时

返回：你发现了什么以及修复了什么的总结。
```

## 常见错误

| 错误做法 | 正确做法 |
|---------|---------|
| 太宽泛："修复所有测试" | 具体明确："修复 agent-tool-abort.test.ts" |
| 无上下文："修复竞态条件" | 提供上下文：粘贴错误信息和测试名称 |
| 无约束：智能体可能重构所有代码 | 设置约束："不要修改生产代码" |
| 模糊的输出要求："修好它" | 明确要求："返回根因和修改内容的总结" |

## 异常处理

| 场景 | 处理策略 |
|------|---------|
| 失败可能相关 | 重新评估关联性；放弃并行改为单一智能体排查 |
| 智能体之间出现冲突 | 1. 检查冲突文件范围和冲突行号<br>2. 优先保留业务逻辑正确的修改<br>3. 手动合并或要求相关智能体重新分配范围 |
| 智能体返回结果质量不足 | 要求补充上下文；转交另一个智能体；单独处理 |
| 资源限制无法并行 | 降级为顺序执行；保持任务聚焦原则 |

> **项目上下文**：当前项目的微服务列表通过 `project-context` 查阅。

## 实际案例

**案例：多服务独立故障排查**

- 服务A：慢SQL导致接口超时
- 服务B：认证偶发失败
- 服务C：文件上传偶发失败

**分派：**
```
智能体1 → 优化服务A慢SQL（通过 `project-context` 查阅数据库设计规范）
智能体2 → 排查服务B认证配置（通过 `project-context` 查阅架构设计规范）
智能体3 → 调整服务C连接池
```

**结果：** 所有修复互相独立，无冲突，各服务压测通过

## 验证 checklist

- [ ] 已确认失败之间确实独立
- [ ] 每个智能体任务范围清晰、自包含
- [ ] 已设置约束条件（不修改其他智能体负责的文件）
- [ ] 已明确预期输出
- [ ] 集成前检查了修改冲突
- [ ] 运行了完整测试套件验证
