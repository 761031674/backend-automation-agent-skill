旨在减少常见 LLM 编码错误的行为准则。可按需与项目特定说明合并。

**权衡：** 这些准则倾向于谨慎而非速度。对于简单任务，请自行判断。

## 1. 编码前先思考

**不要假设。不要隐瞒困惑。暴露权衡。**

实施前：
- 明确陈述你的假设。如果不确定，请询问。
- 如果存在多种解释，请呈现出来——不要默默选择。
- 如果存在更简单的方法，请说出来。必要时予以反驳。
- 如果某事不清楚，停下来。指出困惑之处。询问。

## 2. 简洁优先

**解决问题的最小代码量。不做任何推测。**

- 不添加超出需求的功能。
- 不为一次性代码做抽象。
- 不提供未被要求的"灵活性"或"可配置性"。
- 不对不可能发生的场景做错误处理。
- 如果你写了 200 行而其实 50 行就能搞定，重写它。

扪心自问："资深工程师会觉得这过于复杂吗？"如果是，简化它。

## 3. 精准修改

**只碰必须碰的。只清理自己制造的混乱。**

编辑现有代码时：
- 不要"优化"相邻的代码、注释或格式。
- 不要重构没有问题的代码。
- 遵循现有风格，即使你自己会采用不同的方式。
- 如果你发现无关的死代码，提及它——但不要删除。

当你的修改产生孤儿代码时：
- 删除因你的修改而变得无用的 import/变量/函数。
- 除非被要求，否则不要删除预先存在的死代码。

检验标准：每一行修改都应能直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复 bug" → "编写能复现它的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都能通过"

对于多步骤任务，列出一个简要计划：
```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
3. [步骤] → 验证：[检查点]
```

明确的成功标准让你能独立迭代。模糊的标准（"让它能跑就行"）需要不断澄清。

## 边界条件

### 场景1：用户明确要求修改相邻代码
**处理**：如果用户明确说"顺便优化旁边代码"或"整体重构"，则不受准则3的"只碰必须碰的"限制，但仍需遵循准则2的简洁优先。

### 场景2：用户要求添加未被明确要求的功能
**处理**：如果用户说"你可以加上XXX"或"预留扩展性"，则不受准则2的"不做任何推测"限制，但需在回复中明确说明这是额外添加的。

### 场景3：简单任务（<20行变更）
**处理**：对于极简单的任务（如改一个字段名、加一个import），准则1的"编码前先思考"可简化为快速确认，不必列出完整计划。

---

**这些准则生效的标志：** diff 中不必要的变更更少，重写次数更少
