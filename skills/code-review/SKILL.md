# 代码审查

## 工作流程

1. **理解意图**
   - **输入**：PR标题、变更描述、代码变更文件列表
   - **动作**：阅读描述理解业务意图，统计变更行数，识别涉及的核心模块和外部依赖
   - **输出**：影响范围评估、是否需要拆分的判断
   - **检查点1**：如果代码变更>500行或缺少变更描述，先询问用户是否接受拆分建议或补充上下文，再继续审查
   
2. **分层审查**
   - **输入**：代码diff、相关文件、项目规范（参考reference.md）
   - **动作**：按 安全性→规范性→性能→可维护性 顺序逐层检查，将发现的问题归入🔴/🟡/💭
     - **安全层**：SQL注入（字符串拼接）、XSS（未编码输出）、鉴权绕过、安全框架输入校验
     - **规范层**：实体类继承基础实体类、@Getter/@Setter、@Resource注入、安全框架日志规范、SQL无SELECT *且含逻辑删除字段
     - **性能层**：N+1查询、循环内数据库操作、大对象分页、资源泄漏、长事务
     - **可维护性层**：命名清晰、无魔法值、复杂逻辑有注释、空指针防护
   - **输出**：分级问题清单（含位置、原因、优化方向）
   - **检查点2**：如果阻塞项（🔴）超过3个，询问用户是否优先聚焦阻塞项，还是继续完整审查所有层级
   
   > **用户确认**：在完成每层审查后，简要展示该层发现的问题数量（如"安全层发现2个🔴、1个🟡"），询问"是否继续下一层审查？"
   
3. **输出报告**
   - **输入**：分层审查产生的所有问题、代码中的亮点
   - **动作**：按分级整理问题，补充优化前vs优化后示例，添加开场总结和结尾建议
   - **输出**：完整审查评论（遵循下方「输出格式」和「沟通风格」）
   - **检查点3**：输出前确认是否有遗漏的上下文信息（如相关配置、测试文件、数据库脚本），避免基于不完整信息给出结论

**通用检查点**：意图不明确时提问，而非直接判定为错误；遇到非Java代码或完全空文件时，询问用户审查范围

## 审查前自检清单

开始分层审查前，先完成以下确认：

- [ ] **意图已理解**：能用自己的话概括PR的业务目标和变更动机
- [ ] **范围已确认**：已统计变更文件数和行数，识别出关键路径文件
- [ ] **规范已就绪**：已通过 `project-context` 查阅当前项目审查规范（如 `code-review.md`、`security-audit.md`）
- [ ] **上下文已收集**：相关配置、测试文件、数据库脚本已纳入审查视野（如有）

> **自检结果**：全部勾选 → 进入分层审查；任一未勾选 → 先补充信息再审查

> **审查规范来源**：阻塞项/建议项/小改进的详细清单和代码示例，以及安全审查要点，均通过 `project-context` 查阅。

## 输出格式

每条审查评论必须包含：
1. **分级标记**（🔴/🟡/💭）
2. **问题类别**（安全/规范/性能/可维护性）
3. **具体位置**（文件名和行号）
4. **原因说明**（为什么有问题）
5. **优化示例**（优化前 vs 优化后）

## 关键规则

1. **具体明确** — 说"第42行可能存在SQL注入"，而不是"有安全问题"
2. **解释原因** — 不要只说改什么，要解释为什么
3. **建议而非命令** — 说"可以考虑用X，因为Y"，而不是"改成X"
4. **区分意见和事实** — "这里有内存泄漏"是事实，"我觉得用策略模式更好"是意见
5. **表扬好代码** — 发现巧妙的解决方案要主动肯定
6. **一次到位** — 不要分多轮逐步反馈，一次给出完整意见

## 异常场景处理

| 场景 | 处理策略 |
|------|---------|
| PR规模过大（>500行） | 建议拆分；优先审查关键路径；确认是否接受分阶段审查 |
| 紧急修复（Hotfix） | 聚焦修复正确性；确认安全框架和SQL安全无绕过；建议后续补充测试 |
| 新人代码 | 多解释"为什么"；给出规范链接；肯定做得好的部分；询问是否需要补充文档 |
| 代码完全合规 | 明确肯定优点，不强行挑剔；可给出架构或设计层面的进阶建议 |
| 非Java代码或纯配置变更 | 确认用户是否需要按规范审查；前端代码关注XSS和输入校验；SQL脚本关注索引和回滚方案 |
| 缺少测试文件 | 提醒补充单元测试覆盖关键路径；如为Hotfix可放宽但需记录待补充事项 |
| PR涉及多种技术栈 | 确认各技术栈的审查范围；非Java部分关注安全（XSS/注入）和配置正确性 |
| 审查者与作者存在分歧 | 先确认是事实问题还是意见分歧；事实问题坚持原则，意见分歧给出参考依据并请第三方判断 |

## 审查评论示例

### 🔴 阻塞项示例

```markdown
🔴 **安全 - SQL注入风险**

**位置**：UserMapper.java:42
**问题**：`queryUser` 方法使用字符串拼接构建 SQL
```java
String sql = "SELECT * FROM t_user WHERE name = '" + name + "'";
```
**风险**：攻击者可通过 `name` 参数注入恶意 SQL。
**建议**：使用参数化查询：
```java
@Select("SELECT * FROM t_user WHERE name = #{name}")
User queryUser(@Param("name") String name);
```
```

### 🟡 建议项示例

```markdown
🟡 **性能 - N+1查询**

**位置**：OrderService.java:88
**问题**：循环内逐条查询用户信息
```java
for (Order order : orders) {
    User user = userMapper.selectById(order.getUserId()); // N次查询
}
```
**影响**：100个订单 = 101次查询
**建议**：使用批量查询：
```java
List<Long> userIds = orders.stream().map(Order::getUserId).collect(Collectors.toList());
Map<Long, User> userMap = userMapper.selectBatchIds(userIds).stream()
    .collect(Collectors.toMap(User::getId, Function.identity()));
```
```

### 💭 小改进示例

```markdown
💭 **可维护性 - 魔法值**

**位置**：UserService.java:34
**问题**：硬编码状态值
```java
if (user.getStatus() == 1) { // 1是什么状态？
```
**建议**：使用枚举或常量：
```java
if (user.getStatus() == UserStatus.ACTIVE.getCode()) {
```
```

## 沟通风格

- **开场**：先总结整体印象，指出值得肯定的地方
- **结尾**：以鼓励和下一步建议结尾
- **态度**：建设性、深入、有教育意义、尊重他人
